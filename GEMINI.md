# Gemini CLI システム指示（System Instructions）

あなたは本プロジェクト（広告運用レポート自動化）の開発をサポートする優秀なAIエージェントです。
ツールやセッションをまたいでも文脈を維持し、一貫した品質で成果を出すために、以下の「コンテキストファイル」と「開発ルール」を必ず厳守してください。

## 1. 最優先で読み込むコンテキスト
起動時、またはユーザーから「作業を再開して」と言われた場合、以下のファイルを順に読み込み、現在の状況、ルール、仕様、タスクを完全に把握してください。
1. `.agent/AGENTS.md` （プロジェクトの不変ルール、コーディング規約）
2. `.agent/handoff/HANDOFF.md` （前回のセッションからの引き継ぎ情報）
3. `.agent/memory/MEMORY.md` （これまでに蓄積された教訓・経験）
4. `.spec/` 配下の各ファイル（現在の作業仕様と進捗）

## 2. プロジェクトの制約事項の遵守（最重要）
- 本プロジェクトは**月20時間という非常に厳しい稼働制限**があります。
- 高度で複雑なスクラッチ開発（ヘッドレスブラウザによるPDF化スクレイピング等）は極力避け、**GASやスプレッドシート、Looker Studioの標準機能**を活用したシンプルな解決策を最優先で提案してください。
- 納品後は非エンジニアが運用保守を行うため、コードの可観測性と可読性を高く保ち、複雑なデータ基盤アーキテクチャ（dbtやTerraform等）の導入は避けてください。

## 3. 仕様駆動開発（SDD）ルール
- コーディングや実作業を開始する前に、必ず `.spec/` 配下の4ファイルを確認・更新すること。
- 順序：PLAN（目的確認） → SPEC（要件確認） → TODO（タスク確認） → 実作業
- `PLAN.md` はユーザーの雑多なメモです。これを確認したら、すぐにコードを書かず、不明点をユーザーにヒアリングしながら `SPEC.md` を構造化して作成・更新してください。
- `SPEC.md` が確定したら `TODO.md` にタスクを分解し、ユーザーの承認を得てから実装に入ってください。

## 4. セッション終了時の引き継ぎ（ハンドオフ）ルール
ユーザーからセッション終了（または `/handoff` に類する指示）を求められた場合、あるいは作業の区切りがついた場合は、以下の処理を行ってください。
1. 現在の進捗、次にやるべきこと、未解決の課題を `.agent/handoff/HANDOFF.md` に詳細に書き出してください（既存の内容は上書き）。
2. このセッションで得られた汎用的な教訓やパターンがあれば、`.agent/memory/MEMORY.md` に追記してください。ただし、`AGENTS.md` と重複する内容は書かないこと。
